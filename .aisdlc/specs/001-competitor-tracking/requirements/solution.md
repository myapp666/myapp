# 竞争情报监控系统 - 产品方案

## 0. 基本信息

- 需求标识：`001-competitor-tracking`
- 状态：approved
- 最后更新：2026-07-07
- 关联：`.aisdlc/specs/001-competitor-tracking/requirements/raw.md`

## 1. 结论摘要

- **目标**：构建自动化竞争情报监控系统，替代人工浏览/截图/比对的低效工作流
- **本次 In**：竞对配置 + 定时自动采集 + AI 内容解读 + 历史记录查看；全部通过 `.env` 配置
- **本次 Out**：用户登录/鉴权、实时告警推送、情报收件箱、多角色权限、社媒抓取
- **推荐方案**：`.env` 驱动的轻量单体服务（Python 后端定时采集 + Node.js 前端展示）
- **优先验证点**：V-001（AI 解读质量）、V-002（定时采集稳定性）

## 2. 推荐方案

### 方案名称
`.env` 驱动的竞争情报监控轻量服务（MVP v1）

### 主流程
1. 用户在前端配置竞对信息（名称、官网 URL、行业分类、备注）
2. Python 后端按 `MONITOR_INTERVAL` 定时抓取 `MONITOR_URLS` 中各目标网站
3. 抓取内容调用 `AI_API_KEY` 对应 LLM，生成结构化变化解读
4. 解读结果存入数据库，附带时间戳与原始快照
5. 前端展示每个竞对的历史记录列表，支持按时间查看变化详情

### 关键边界与取舍
1. **无用户系统**：MVP 不做登录/鉴权，内部事业部直接访问
2. **配置全在 `.env`**：`MONITOR_INTERVAL`、`MONITOR_URLS`、`AI_API_KEY` 均通过环境变量管理
3. **字段精简**：竞对配置仅 4 个字段（name、website_url、industry、notes）
4. **历史记录只读**：前端只支持查看，不支持编辑或人工标注
5. **单一 AI 接口**：MVP 阶段只接入一个 LLM，不做多模型路由

### 为什么选它
1. **最快可验证**：`.env` 驱动省去了后台管理页面，周期可控制在 1-2 周内完成内测版
2. **与技术栈契合**：Python 擅长定时任务和网络请求，Node.js 适合轻量前端展示
3. **容易迭代**：无权限系统、无复边界清晰，后续 v2 可按需叠加

## 3. 备选方案

### 3.A 最小版（仅配置 + 手动查询，无自动采集）
- 核心机制：用户手动输入竞对信息，后台简单列表展示
- 不选原因：用户明确要求定时采集 + AI 解读，该方案无法验证核心价值

### 3.B 全功能版（含用户系统 + 推送 + 多数据源）
- 核心机制：完整 SaaS 架构，含登录、角色权限、Slack 推送、社媒抓取
- 不选原因：开发周期 2-3 个月，核心需求未验证，风险过高；不符合分阶段策略

### 3.C 第三方服务集成（如 Brandwatch、Crayon）
- 核心机制：直接购买商业情报工具
- 不选原因：成本高、定制性差、无法沉淀自有数据资产

## 5. 验证清单

### V-001：AI 解读质量
- **假设**：LLM 能从网站 HTML 变化中提取有意义的竞品动态（如产品更新、定价变化）
- **验证方法**：人工选取 3-5 个已知有变化的竞对页面，对比 AI 解读与人工判断的吻合度
- **成功信号**：80% 以上的变化被正确识别且解读有参考价值
- **失败信号**：AI 频繁输出"无变化"或解读内容与实际不符
- **Owner**：开发 + 产品经理
- **截止**：内测第 1 周
- **触发动作**：若不达标，调整 prompt 策略或更换 LLM 模型

### V-002：定时采集稳定性
- **假设**：Python 定时任务能稳定运行，不因网站反爬或网络问题频繁失败
- **验证方法**：部署后连续运行 7 天，统计采集成功率
- **成功信号**：采集成功率 > 90%
- **失败信号**：成功率 < 70%，或频繁 IP 封禁
- **Owner**：开发
- **截止**：内测第 2 周
- **触发动作**：若不达标，引入代理池或降低采集频率

### V-003：用户体验验收
- **假设**：前端历史记录页面能清晰展示竞对变化，用户可在 2 分钟内完成查阅任务
- **验证方法**：邀请 2-3 位内测用户完成"查找某竞对近 7 天变化"任务，记录时长
- **成功信号**：所有用户在 2 分钟内完成，无需帮助
- **失败信号**：超过一半用户需要指引或反馈界面不清晰
- **Owner**：产品经理
- **截止**：内测第 2 周
- **触发动作**：若不达标，优化前端布局与信息层级

### V-004：`.env` 配置可维护性
- **假设**：运维人员能通过 `.env` 独立完成监控目标的增删改，无需修改代码
- **验证方法**：由非开发人员按文档操作，增加 1 个竞对 URL 并验证下一采集周期正确抓取
- **成功信号**：全程无需开发介入，操作成功
- **失败信号**：需要修改代码或报错无法自行排查
- **Owner**：开发（提供文档）+ 运维
- **截止**：内测第 1 周
- **触发动作置说明文档或改为简单后台配置页

## 6. Context Gaps

| 缺口 | 影响 | 验证/缓解方案 |
|---|---|---|
| `CONTEXT GAP`：现有 myapp 为纯静态页面，无后端 | 需评估是否同仓库扩展或新建仓库 | 设计阶段确认仓库结构 |
| `CONTEXT GAP`：目标网站反爬策略未知 | 直接影响采集稳定性 | V-002 验证；预留代理池方案 |
| `CONTEXT GAP`：LLM 选型未确定 | AI 解读质量依赖模型能力 | V-001 验证；`.env` 中 `AI_API_KEY` 可替换 |
| `CONTEXT GAP`：数据库选型未定 | 影响历史记录存储与查询性能 | 建议 SQLite（MVP）或 PostgreSQL（生产） |
| `CONTEXT GAP`：前后端通信方式未定 | Python 后端与 Node.js 前端的接口设计 | 建议 REST API；设计阶段确认 |

## 7. Impact Analysis

### 7.1 受影响模块

| 模块 | 影响类型 | 关键不变量 | stale? |
|---|---|---|---|
| 竞争情报监控系统（新建） | 新增服务 | 独立服务，无现有依赖 | no |
| myapp 前端 | 扩展/新增页面 | 复用现有 Node.js 技术栈 | no |
| Python 后端（新建） | 新增服务 | 定时任务 + REST API | no |

### 7.2 需遵守的不变量
1. **数据隐私**：竞对信息仅限内部访问，不对入数据库 → Node.js 前端通过 REST API 读取
- 若后续（v2）引入推送：需新增通知服务模块，不影响现有采集逻辑

### 7.4 Context Gaps（详见第 6 节）
- 现有 myapp 为纯静态页面，需设计阶段确认仓库结构
- 前后端通信方式（REST API vs 共享 DB）待设计阶段确认
