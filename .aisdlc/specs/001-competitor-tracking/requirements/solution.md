# 竞争情报监控系统 - 产品方案

## 0. 基本信息

- 需求标识：`001-competitor-tracking`
- 状态：approved
- 最后更新：2026-07-07
- 关联：`.aisdlc/specs/001-competitor-tracking/requirements/raw.md`

## 1. 结论摘要

- **目标**：构建自动化竞争情报监控系统，替代人工浏览/截图/比对的低效工作流
- **本次 In**：用户注册/登录（JWT）+ 按用户隔离的竞对配置 + 定时自动采集 + AI 内容解读 + 历史记录查看
- **本次 Out**：忘记密码/邮件重置、第三方 OAuth、多角色权限、实时告警推送、情报收件箱、社媒抓取
- **推荐方案**：SQLite 承载用户账号与按用户隔离的竞对配置 + JWT 鉴权 + Python 后端定时采集 + Node.js 前端展示
- **优先验证点**：V-001（AI 解读质量）、V-002（定时采集稳定性）、V-005（快照存储方案裁决）

## 2. 推荐方案

### 方案名称
JWT 登录 + SQLite 用户隔离的竞争情报监控轻量服务（MVP v2）

### 主流程
1. 用户通过邮箱+密码在前端完成注册/登录，获得 access token（24h）+ refresh token（7 天）
2. 登录后用户在前端配置自己的竞对信息（名称、官网 URL、行业分类、备注），存入 SQLite（关联 `user_id`）
3. Python 后端按 `MONITOR_INTERVAL`（`.env` 全局配置）定时抓取所有用户配置的目标网站
4. 抓取内容调用 `AI_API_KEY`（`.env` 全局配置）对应 LLM，生成结构化变化解读
5. 解读结果存入数据库，附带时间戳、原始快照与所属竞对配置的关联
6. 前端携带 token 调用 REST API，只能查看当前用户自己配置的竞对历史记录

### 关键边界与取舍
1. **引入用户系统**：邮箱+密码注册/登录，JWT 无状态鉴权；无第三方 OAuth、无多角色权限
2. **数据隔离**：竞对配置与历史记录均关联 `user_id`，用户之间数据互不可见
3. **配置分层**：`MONITOR_INTERVAL`、`AI_API_KEY` 仍走 `.env`（全局、不涉及用户隔离）；竞对清单从 `.env` 迁移到 SQLite（按用户维护，不做旧数据迁移，MVP 直接切换）
4. **字段精简**：竞对配置仍为 4 个字段（name、website_url、industry、notes）+ `user_id`
5. **历史记录只读**：前端只支持查看，不支持编辑或人工标注
6. **单一 AI 接口**：MVP 阶段只接入一个 LLM，不做多模型路由
7. **密码重置 Out**：忘记密码走人工干预（数据库手动重置），不集成邮件服务

### 为什么选它
1. **满足新增需求且改动可控**：仅需新增用户表、竞对配置表迁移到 SQLite，采集/AI 解读逻辑不变
2. **与技术栈契合**：JWT 无状态鉴权天然适配 Vercel（前端）+ 独立后端部署的分离架构，无需引入会话存储组件
3. **容易迭代**：数据隔离模型清晰，后续 v2 若要加角色权限或组织级共享，可在现有 `user_id` 外键基础上扩展

## 3. 备选方案

### 3.A 最小版（仅配置 + 手动查询，无自动采集）
- 核心机制：用户手动输入竞对信息，后台简单列表展示
- 不选原因：用户明确要求定时采集 + AI 解读，该方案无法验证核心价值

### 3.B 全功能版（含多角色权限 + OAuth + 推送 + 多数据源）
- 核心机制：完整 SaaS 架构，含第三方 OAuth 登录、角色权限体系、Slack 推送、社媒抓取
- 不选原因：开发周期 2-3 个月，核心需求未验证，风险过高；MVP 邮箱密码登录已能满足内测隔离需求，不符合分阶段策略

### 3.C 第三方服务集成（如 Brandwatch、Crayon）
- 核心机制：直接购买商业情报工具
- 不选原因：成本高、定制性差、无法沉淀自有数据资产

### 3.D 服务端 Session + Cookie 鉴权（登录方案备选）
- 核心机制：后端维护会话状态（内存或 Redis），前端依赖 Cookie 自动携带 session id
- 不选原因：Next.js 前端部署在 Vercel、FastAPI 后端独立部署，跨域场景下 Cookie 配置更复杂；且需额外引入会话存储组件，与当前无状态服务架构不契合（已在第 12 轮澄清中确认选择 JWT）

## 5. 验证清单

### V-001：AI 解读质量
- **假设**：LLM 能从网站 HTML 变化中提取有意义的竞品动态（如产品更新、定价变化）
- **验证方法**：人工选取 3-5 个已知有变化的竞对页面，对比 AI 解读与人工判断的吻合度
- **成功信号**：80% 以上的变化被正确识别且解读有参考价值
- **失败信号**：AI 频繁输出"无变化"或解读内容与实际不符
- **Owner**：开发 + 产品经理
- **截止**：内测第 1 周
- **触发动作**：若不达标，调整 prompt 策略或更换 LLM 模型

### V-002：定时采集稳定性
- **假设**：Python 定时任务能稳定运行，不因网站反爬或网络问题频繁失败
- **验证方法**：部署后连续运行 7 天，统计采集成功率
- **成功信号**：采集成功率 > 90%
- **失败信号**：成功率 < 70%，或频繁 IP 封禁
- **Owner**：开发
- **截止**：内测第 2 周
- **触发动作**：若不达标，引入代理池或降低采集频率

### V-003：用户体验验收
- **假设**：前端历史记录页面能清晰展示竞对变化，用户可在 2 分钟内完成查阅任务
- **验证方法**：邀请 2-3 位内测用户完成"查找某竞对近 7 天变化"任务，记录时长
- **成功信号**：所有用户在 2 分钟内完成，无需帮助
- **失败信号**：超过一半用户需要指引或反馈界面不清晰
- **Owner**：产品经理
- **截止**：内测第 2 周
- **触发动作**：若不达标，优化前端布局与信息层级

### V-004：竞对配置可维护性（用户自助）
- **假设**：登录用户能通过前端页面独立完成竞对的增删改，无需开发人员介入或修改代码
- **验证方法**：由非开发人员登录后按操作文档新增 1 个竞对，并验证下一采集周期正确抓取
- **成功信号**：全程无需开发介入，操作成功
- **失败信号**：需要修改代码或报错无法自行排查
- **Owner**：开发（提供文档）+ 运维
- **截止**：内测第 1 周
- **触发动作**：不达标则改写操作说明文档或简化前端表单交互

### V-005：采集快照存储方案裁决
- **假设**：引入 SQLite 承载用户账号与竞对配置后，采集快照（snapshot）是否也迁移到 SQLite，或继续用 JSON 文件 + filelock，尚需设计阶段裁决
- **验证方法**：设计阶段（D 链路）评估两种方案在查询性能（按 user_id 过滤历史记录）、并发写入安全性上的差异
- **成功信号**：设计阶段产出明确结论并写入 `design/design.md`
- **失败信号**：设计阶段未给出结论，实现阶段仍存在歧义
- **Owner**：开发
- **截止**：设计阶段（D 链路）完成前
- **触发动作**：若设计阶段未决，默认沿用 JSON 文件 + filelock，按 `user_id` 分目录/分文件存储

### V-006：JWT secret 管理与 token 存储安全
- **假设**：JWT secret 通过 `.env` 管理且强度足够（≥32 字节随机串），前端 token 存储方式不引入 XSS 风险
- **验证方法**：代码审查确认 secret 来源与强度；审查前端 token 存储方式（httpOnly cookie 或 localStorage + 防 XSS 措施）
- **成功信号**：审查通过，无高危发现
- **失败信号**：secret 硬编码或强度不足；token 存储方式存在明显 XSS 风险
- **Owner**：开发
- **截止**：MVP 部署前
- **触发动作**：不达标则改用 httpOnly cookie 存储 token，并将 secret 迁移到密钥管理机制（v2）

## 6. Context Gaps

| 缺口 | 影响 | 验证/缓解方案 |
|---|---|---|
| `CONTEXT GAP`：现有 myapp 为纯静态页面，无后端 | 需评估是否同仓库扩展或新建仓库 | 设计阶段确认仓库结构 |

| `CONTEXT GAP`：JWT secret 生成与管理方式未落地 | 影响鉴权安全性 | V-006 验证；设计阶段确定 `.env` 注入方式与 secret 生成脚本 |

## 7. Impact Analysis

### 7.1 受影响模块

| 模块 | 影响类型 | 关键不变量 | stale? |
|---|---|---|---|
| 竞争情报监控系统（新建） | 新增服务 | 独立服务，无现有依赖 | no |
| myapp 前端 | 扩展/新增页面（含登录/注册页） | 复用现有 Node.js 技术栈 | no |
| Python 后端（新建） | 新增服务 | 定时任务 + REST API + JWT 鉴权 | no |
| 用户账号与竞对配置存储（新建） | 新增 SQLite 数据层 | 替代原 `.env` 全局 `MONITOR_URLS`；按 `user_id` 隔离 | no |

### 7.2 需遵守的不变量
1. **数据隐私**：竞对信息与历史记录仅限所属用户访问，不跨用户可见 → REST API 按 `user_id` 过滤，前端不直接访问数据库
2. **密码安全**：密码只存 bcrypt hash，不落盘明文、不出现在日志
3. **无状态鉴权**：后端不维护会话存储，鉴权状态完全由 JWT 承载，兼容 Vercel 前端 + 独立后端部署架构

### 7.3 跨模块影响
- 改了竞对配置存储方式（`.env` → SQLite） → 需关注采集调度模块（原按 `.env` 的 `MONITOR_URLS` 读取全部 URL，改为按 SQLite 查询所有用户的竞对配置汇总后调度）
- 改了鉴权模型（新增用户系统） → 需关注前端所有调用 `/api/*` 的位置，统一补充 Bearer token 携带逻辑
- 若后续（v2）引入推送：需新增通知服务模块，且推送目标需按用户区分，不影响现有采集逻辑

### 7.4 Context Gaps（详见第 6 节）
- 现有 myapp 为纯静态页面，需设计阶段确认仓库结构
- 前后端通信方式（REST API vs 共享 DB）待设计阶段确认
- 采集快照存储方式（SQLite vs JSON 文件）待设计阶段裁决（V-005）
- JWT secret 管理方式待设计阶段落地（V-006）

- 扩展 MVP 范围：新增定时自动采集、AI 内容解读、历史记录查看

### v3.0（2026-07-07）
- 推翻 v2.0 第 6 轮"无权限系统"决策：新增用户注册/登录（邮箱+密码 + JWT 无状态鉴权）

- 注册范围：任意邮箱开放注册，不设域名白名单/邀请码
- 忘记密码功能列为 MVP Out，人工干预（数据库手动重置）替代
- 新增 V-005（快照存储方案裁决）、V-006（JWT secret 管理与 token 安全）验证项
- 新增备选方案 3.D（服务端 Session + Cookie），说明为何选 JWT 而非该方案
- 原因：用户明确要求"添加用户登录"功能，经 8 轮澄清（raw.md 第 8–13 轮）确定登录范围与数据隔离模型
- 决策点：数据隔离粒度 → 按用户；鉴权机制 → JWT；存储方案 → 引入 SQLite；密码重置 → MVP Out
