在快速变化的市场中，团队需要持续跟踪竞争对手的动向（如产品迭代、定价调整、营销活动、舆情变化等），以辅助自身的产品、市场与商业决策。目前这项工作主要依赖人工浏览、截图与比对，耗时、低频、易遗漏，且难以沉淀为结构化情报。

## 澄清记录

### 第 1 轮：监控范围与更新频率
**本轮结论**：
- 采用**高定制方案**：动态配置竞对清单 + 自定义数据源 + 实时告警
- 不限制竞对数量；支持多数据源接入

**本轮新增约束**：
1. 系统应支持灵活的竞对配置（不硬编码）
2. 数据源应可扩展（官网爬虫、API、人工输入）
3. 告警机制应支持实时或近实时

**关键决策**：范围：动态竞对清单 + 自定义数据源；频率：实时告警

### 第 2 轮：用户角色与消费形式
**本轮结论**：
- 用户包括 PM、市场、创业者、投资人
- 核心需求是**快速建立监控、自动采集解读、结构化消费、可决策建议、主动通知**
- 7 大用户需求已明确（UR-1 到 UR-7）

**本轮新增约束**：
1. 情报必须包含"发生了什么 - 为什么重要 - 我该怎么做"三层解读
2. 系统应主动推送（邮件/Slack），而非被动查询
3. 用户反馈机制能驱动质量改进

### 第 3 轮：MVP 优先级
**本轮结论**：
- 采用**分阶段 MVP 策略**
- 第一阶段（内测版）：竞对定义 + 手动/API 配置数据源 + 简单后台查询
- 范围明确，验收清晰

**本轮新增约束**：
1. MVP 不包含自动采集、AI 解读、情报收件箱、实时告警
2. MVP 用于事业部内测验证需求
3. 后续迭代补齐完整功能

### 第 4 轮：技术栈
**本轮结论**：
- 前端：Node.js 技术栈
- 后端：Python 定时采集
- 不使用专项 LLM 框架；继续沿用现有技术方向

**本轮新增约束**：
1. 前后分离架构
2. Python 后端负责定时任务与数据采集逻辑
3. Node.js 前端负责用户界面与交互

### 第 5 轮：竞对配置字段

**本轮结论**：
- MVP 竞对配置页采用**基础字段**：名称、官网 URL、行业分类、备注

**本轮新增约束**：
1. MVP 数据模型字段：`name`、`website_url`、`industry`、`notes`
2. 字段设计保持轻量，便于后续扩展（产品线、定价区间、社媒账号等）

**关键决策**：字段 → 基础字段（先内测验证，再按需扩展）

### 第 6 轮：权限与配置管理

**本轮结论**：
- MVP 不做用户权限系统
- 配置（API Key、数据源等）放在 `.env` 文件中由 Python 后端读取

**本轮新增约束**：
1. 无用户登录 / 鉴权需求（MVP 阶段）
2. 敏感配置通过 `.env` 管理，不存入数据库
3. Python 后端通过环境变量读取配置

**关键决策**：权限 → 无权限系统；配置 → `.env` 文件

### 第 7 轮：核心功能补充

**本轮结论**：
- `.env` 配置项扩展：监控间隔、监控网站列表、AI API 配置
- 前端新增：每个竞对网站的历史记录查看页

**本轮新增约束**：
1. `MONITOR_INTERVAL`（监控间隔，单位分钟/小时）通过 `.env` 配置
2. `MONITOR_URLS`（监控网站列表）通过 `.env` 配置（支持多个）
3. AI API Key（如 OpenAI/Claude）通过 `.env` 配置，用于内容解读
4. 前端需展示每个监控网站的**历史变化记录**，支持按时间浏览
5. Python 后端按间隔定时抓取，结果存库，前端查询展示

**关键决策**：
- MVP 包含自动采集（定时爬取）+ AI 解读 + 历史记录查看
- 这扩展了原 MVP v1 范围，需更新 solution.md

### 第 8 轮：引入用户登录与数据隔离模型

**本轮结论**：
- 推翻第 6 轮"MVP 不做用户权限系统"的决策：MVP 新增用户注册/登录（邮箱+密码）
- 登录后采用**按用户隔离数据**：每个用户只能看到自己配置的竞对和对应的历史记录

**本轮新增约束**：
1. 竞对配置（name/website_url/industry/notes）需关联 `user_id`，不同用户各自维护独立列表
2. 历史采集记录（snapshot）需可追溯到所属用户的竞对配置
3. 原本通过 `.env` 的 `MONITOR_URLS` 全局配置方式不再适用于多用户场景，竞对配置来源需要从"全局 `.env`"变为"按用户的持久化配置"（具体存储方式待设计阶段确定，通过验证清单跟踪）
4. `.env` 中仍保留 `MONITOR_INTERVAL`（全局采集间隔）与 `AI_API_KEY`（全局 AI 配置），这两项不涉及用户隔离
5. FastAPI 查询接口（`/api/snapshots`、`/api/urls`）需按当前登录用户过滤，不能返回其他用户的数据

**关键决策**：权限 → 邮箱+密码登录 + JWT 鉴权；数据隔离 → 按用户隔离（竞对配置与历史记录均归属用户）；`.env` 全局配置 → 保留采集间隔与 AI Key，竞对清单迁移至按用户存储

**遗留歧义**：
- 假设/风险：竞对配置从 `.env` 迁移到按用户存储的具体方案（数据库选型、迁移路径）尚未确定 → 对应验证 V-005

### 第 9 轮：`.env` 遗留竞对配置的迁移策略

**本轮结论**：
- 不做数据迁移，MVP 直接切换：废弃 `.env` 的 `MONITOR_URLS`，上线即要求所有用户通过前端重新添加竞对配置

**本轮新增约束**：
1. `MONITOR_URLS` 环境变量在切换后失效，不再被后端读取
2. 无需开发迁移脚本，无需定义"首个用户即管理员"等特殊角色
3. 内测阶段数据量小，重新配置成本低，此约束为一次性代价，不影响后续迭代

**关键决策**：`.env` 遗留配置 → 不迁移，直接切换到按用户配置模式

### 第 10 轮：注册开放范围

**本轮结论**：
- 任意邮箱均可注册，不做域名白名单或邀请码限制

**本轮新增约束**：
1. `/auth/register` 不校验邮箱域名，仅校验邮箱格式合法性与唯一性
2. MVP 阶段无需邀请码/白名单机制；内测范围通过访问地址的传播范围自然控制
3. 若后续需要收紧注册范围，作为 v2 增量功能（域名白名单或邀请码）追加，不影响当前数据模型

**关键决策**：注册范围 → 任意邮箱开放注册，无额外限制

### 第 11 轮：引入用户账号后的存储方案

**本轮结论**：
- 引入 SQLite：用户账号表、竞对配置表（带 `user_id` 外键）存入 SQLite，替代原本 `.env` 全局配置
- 采集快照（snapshot）存储方式（继续用 JSON 文件，或一并迁移到 SQLite）留给设计阶段决定，不在本轮固化

**本轮新增约束**：
1. 新增 `users` 表：邮箱（唯一索引）、密码 bcrypt hash、创建时间
2. 新增竞对配置表：`user_id` 外键 + `name`/`website_url`/`industry`/`notes`
3. `.env` 中 `MONITOR_INTERVAL`、`AI_API_KEY` 保持环境变量管理，不进入 SQLite
4. 原 JSON 文件 + filelock 方案（`backend/data/{url_slug}.json`）用于快照存储的部分，是否迁移到 SQLite 由设计阶段（D 链路）评估决定

**关键决策**：存储方案 → 引入 SQLite 承载用户账号与竞对配置；快照存储方式留给设计阶段裁决（对应验证 V-005）

### 第 12 轮：鉴权机制确认

**本轮结论**：
- 确认采用 JWT 无状态鉴权（与已起草的 PRD v2.0 一致）：access token（24h）+ refresh token（7 天），前端通过 Bearer token 调用后端 API

**本轮新增约束**：
1. 后端不维护会话存储（无需 Redis/内存 session），符合 FastAPI 无状态服务 + Vercel 前端分离部署的架构
2. token 校验通过 FastAPI 依赖注入统一处理

**关键决策**：鉴权机制 → JWT 无状态鉴权（access token 24h + refresh token 7 天）

### 第 13 轮：忘记密码功能范围

**本轮结论**：
- MVP 不实现邮件重置密码流程；忘记密码时由开发人员通过数据库/后台手动重置

**本轮新增约束**：
1. `/auth/*` 接口范围内不包含 `forgot-password`/`reset-password`
2. 无需集成邮件发送服务（SMTP/SendGrid 等）
3. 此约束为 MVP Out，明确写入 solution.md 与 prd.md 的 Out 边界，避免后续误判为遗漏

**关键决策**：忘记密码 → MVP Out，人工干预替代
